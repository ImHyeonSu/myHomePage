# agile

# アジャイル / スクラム / スプリント

---

## 三つの関係

```
アジャイル       → 哲学 / 思想
  └─ スクラム        → 実践方法論
       └─ スプリント      → 実際の開発サイクル単位
```

---

## アジャイル (Agile)

> **「完璧な計画より、早く作って早く直そう」**

従来のウォーターフォール方式（要件定義 → 設計 → 開発 → リリース、約1年）は、全部作ってからお客さんに見せると「これじゃないんですけど」となる問題があった。そこで2001年に生まれた開発哲学。

**核心的な価値：**

- 膨大なドキュメントより → **動くソフトウェア**
- 計画に従うより → **変化への対応**
- 最初だけでなく → **顧客との継続的な協力**

---

## スクラム (Scrum)

> **アジャイルをどう実践するかの具体的なフレームワーク**

### 役割

| 役割              | やること                                   |
| ----------------- | ------------------------------------------ |
| **Product Owner** | 何を作るか決める、優先順位の管理           |
| **Scrum Master**  | チームがうまく回るようサポート、障害の除去 |
| **開発チーム**    | 実際に作る人たち、自律的にタスクを管理     |

### 成果物

- **Product Backlog** : 作るべき機能の全リスト（PO管理）
- **Sprint Backlog** : 今スプリントでやることだけ抜き出したリスト
- **Increment** : スプリント終了時に出来上がる動作する成果物

---

## スプリント (Sprint)

> **実際の開発が回る1〜2週間の繰り返しサイクル**

```
[計画] → [開発] → [レビュー] → [振り返り] → 次のスプリントへ
```

| イベント               | タイミング | 内容                                 |
| ---------------------- | ---------- | ------------------------------------ |
| **スプリント計画**     | 初日       | 今回何を作るか決める                 |
| **デイリースクラム**   | 毎日15分   | 昨日/今日/詰まっていること共有       |
| **スプリントレビュー** | 最終日     | 作ったものをデモ、フィードバック収集 |
| **振り返り**           | 最終日     | よかった点 / 改善点の議論            |

### 重要な原則

- スプリント中に範囲を変えない
- 終わったら必ず**動く成果物**を出す
- 失敗してもいい。2週間ごとに学んで改善するのが目的
